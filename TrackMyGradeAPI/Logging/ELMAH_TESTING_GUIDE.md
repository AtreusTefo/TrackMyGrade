# ELMAH Testing and Verification Guide

This guide provides step-by-step instructions to test and verify that ELMAH error logging is properly integrated into the TrackMyGrade API.

## Quick Test via Test Endpoints

The API includes test endpoints specifically designed to trigger error logging without affecting production data.

### 1. Health Check

Verify the API is running and ELMAH is accessible:

```powershell
Invoke-WebRequest -Uri "http://localhost:5000/api/test/health" -Method Get
```

**Expected Response:**
```json
{
  "status": "healthy",
  "timestamp": "2026-05-18T20:00:00.0000000Z",
  "message": "TrackMyGrade API is running. ELMAH error logging is enabled."
}
```

### 2. Basic Error Logging Test

Trigger a simple exception to verify ELMAH is capturing errors:

```powershell
Invoke-WebRequest -Uri "http://localhost:5000/api/test/throw-error" -Method Get
```

**Expected Response:** HTTP 500 (Internal Server Error)
**ELMAH Log:** Check the MemoryErrorLog to verify the error was captured with:
- Exception type: `InvalidOperationException`
- Message: "This is a test exception generated by..."
- Context: Custom wrapper message "TEST: Intentional error to verify ELMAH logging"

### 3. Error with Full Context Test

Trigger an error with additional context information:

```powershell
# First, get an auth token (if needed, or use anonymous access)
Invoke-WebRequest -Uri "http://localhost:5000/api/test/throw-error-with-context" -Method Get
```

**ELMAH Log Verification:** Check that the error includes:
- Context data (UserId, RequestUri, Method, ContentType)
- Custom data (TestEndpoint, TestPurpose)
- All context variables prefixed with `CONTEXT_`

### 4. Data Integrity Error Test

Test logging of data consistency errors:

```powershell
Invoke-WebRequest -Uri "http://localhost:5000/api/test/throw-data-integrity-error" -Method Get
```

**ELMAH Log Verification:** Check that the error includes:
- Exception type: `DBConcurrencyException`
- Context: Operation (UPDATE), EntityType (Student), EntityId (123)
- Message: "Data integrity violation during UPDATE of Student"

## Accessing the Error Log

### Development (In-Memory Storage)

ELMAH currently uses `MemoryErrorLog` which stores errors in memory. Errors are lost when the application restarts.

**Note:** The web-based viewer (`/elmah.axd`) only works when deployed under IIS. In self-hosted OWIN mode, use the programmatic API or switch to persistent storage.

### Programmatic Access (Development)

```csharp
using Elmah;

// Retrieve all logged errors
var errorLog = ErrorLog.GetDefault(null);
var errors = errorLog.GetErrors(0, 10);  // Get first 10 errors

foreach (var error in errors)
{
	Console.WriteLine($"[{error.Time}] {error.Exception.Type}: {error.Message}");
}
```

### Persistent Storage (Recommended for Testing/Production)

Switch from `MemoryErrorLog` to `XmlFileErrorLog` for persistent error storage:

**1. Update App.config:**

```xml
<elmah>
  <security allowRemoteAccess="0"/>
  <!-- Change from MemoryErrorLog to XmlFileErrorLog -->
  <errorLog type="Elmah.XmlFileErrorLog, Elmah" logPath="App_Data\errors"/>
</elmah>
```

**2. Create the App_Data directory:**

```powershell
New-Item -ItemType Directory -Path "TrackMyGradeAPI\App_Data\errors" -Force
```

**3. Restart the application**

**4. Run test endpoints again**

**5. View logged errors:**

```powershell
Get-ChildItem -Path "TrackMyGradeAPI\App_Data\errors" -Filter "*.xml" | Sort-Object LastWriteTime -Descending | Select-Object -First 5
```

Each file is an XML error report with full exception details, stack traces, and context.

## End-to-End Testing Scenarios

### Scenario 1: Controller Exception Handling

**What to verify:** Controllers properly catch exceptions and log them to ELMAH

**Test:**
1. Call an API endpoint that triggers a service error
2. Verify ELMAH receives the exception
3. Verify the error includes context (UserId, RequestUri, Method)

**Example:** POST to `/api/admin/teachers` with invalid data

```powershell
$body = @{
	Email = "invalid-email"  # Invalid email format
	FirstName = ""          # Empty first name
	LastName = ""           # Empty last name
} | ConvertTo-Json

Invoke-WebRequest -Uri "http://localhost:5000/api/admin/teachers" `
  -Method Post `
  -Headers @{"X-TeacherId" = "1"} `
  -ContentType "application/json" `
  -Body $body
```

**Expected:** 400 Bad Request, error logged to ELMAH

### Scenario 2: Web API Pipeline Exception

**What to verify:** Unhandled Web API exceptions are caught by ElmahExceptionLogger

**Test:**
1. Call `/api/test/throw-error`
2. Verify exception is caught by Web API pipeline
3. Verify ElmahExceptionLogger is invoked
4. Verify error appears in ELMAH log

### Scenario 3: Startup Exception

**What to verify:** Exceptions during application startup are logged to ELMAH

**Test:**
1. Stop the application
2. Introduce a configuration error (e.g., invalid connection string in App.config)
3. Start the application
4. Observe console output showing startup exception
5. Verify error was logged via ErrorLoggingConfig.LogErrorWithMessage()

### Scenario 4: Middleware Exception

**What to verify:** Unhandled exceptions in OWIN pipeline are caught by ErrorHandlingMiddleware

**Test:**
1. Call an endpoint that would trigger an uncaught middleware error
2. Verify ErrorHandlingMiddleware catches the exception
3. Verify error is logged with full request context
4. Verify error includes middleware metadata ("MiddlewareLevel": "ErrorHandlingMiddleware")

### Scenario 5: Exception Chain Logging

**What to verify:** Nested exceptions (InnerException) are properly captured and chainable

**Test:**
1. Call `/api/test/throw-error` to trigger a wrapped exception
2. Inspect ELMAH error log for the logged error
3. Verify the error includes:
   - **Wrapper Message:** "TEST: Intentional error to verify ELMAH logging"
   - **Inner Exception:** Original InvalidOperationException
   - **ExceptionChainDepth:** Server variable showing depth (e.g., 2 for wrapped exception)
   - **Stack Trace:** Full call stack from both wrapper and original exception

**Expected ELMAH ServerVariables:**
- `ExceptionChainDepth` = 2 (wrapper + original)
- `CONTEXT_*` variables (all custom context properties)

**Code Example of Exception Chain:**
```csharp
try
{
    throw new InvalidOperationException("Root cause error");
}
catch (Exception ex)
{
    // This wraps the exception
    ErrorLoggingConfig.LogErrorWithMessage("Top-level wrapper message", ex);
    throw;
}
```

**Result in ELMAH:**
- Message: "Top-level wrapper message"
- InnerException Type: InvalidOperationException
- InnerException Message: "Root cause error"
- Full chain visible in ELMAH viewer

## Verification Checklist

Use this checklist to verify complete ELMAH integration:

### Configuration
- [ ] App.config has ELMAH `<configSections>` defined
- [ ] App.config has `<elmah>` section with errorLog type
- [ ] Web.config (if IIS deployed) has ELMAH modules and handlers
- [ ] Connection string matches the errorLog type (if using SQL Server)

### Code
- [ ] ErrorLoggingConfig.cs exists with logging methods
- [ ] ElmahExceptionHandler.cs is registered in WebApiConfig
- [ ] ElmahExceptionLogger.cs is registered in WebApiConfig
- [ ] ErrorHandlingMiddleware.cs exists and is registered in Startup.cs
- [ ] Program.cs Main() logs startup exceptions
- [ ] All controllers use ErrorLoggingConfig.LogError() in catch blocks

### Runtime
- [ ] API starts without errors
- [ ] /api/test/health returns 200 OK
- [ ] /api/test/throw-error returns 500 and logs error
- [ ] Error context includes UserId, RequestUri, Method, ContentType
- [ ] Exception chain depth is captured
- [ ] Errors are visible in ELMAH log storage

### Error Scenarios
- [ ] Controller-level exceptions are logged
- [ ] Web API pipeline exceptions are logged
- [ ] Startup exceptions are logged
- [ ] Middleware-level exceptions are logged
- [ ] Validation errors include entity/field context
- [ ] Data integrity errors include operation/entity context

## Troubleshooting

### Test endpoints return 404

**Issue:** `/api/test/*` endpoints not found

**Solution:** 
- Verify TestController.cs is compiled and deployed
- Verify the route prefix is correct: `[RoutePrefix("api/test")]`
- Verify Web API attribute routing is enabled in WebApiConfig.cs

### Errors not appearing in ELMAH log

**Issue:** Exceptions are thrown but don't appear in ELMAH

**Solutions:**
1. Verify ErrorLoggingConfig.InitializeErrorLogging() is called in Startup.cs
2. Verify ErrorLog.GetDefault(null).Log() is being invoked
3. Check App.config has valid errorLog section
4. Verify exception is not filtered by errorFilter rules
5. If using SQL Server, verify connection string and database tables exist
6. Check application event logs for ELMAH initialization errors

### MemoryErrorLog loses errors on restart

**Issue:** Logged errors disappear when application restarts

**Solution:** Switch to persistent storage (XmlFileErrorLog or SqlErrorLog) as described in "Persistent Storage" section above.

## Removing Test Endpoints (Production)

Before deploying to production, remove or restrict the test endpoints:

**Option 1: Remove completely**
```powershell
Remove-Item -Path "TrackMyGradeAPI\Presentation\Controllers\TestController.cs"
```

**Option 2: Restrict to development**
Add a conditional compilation or authorization check:
```csharp
#if DEBUG
[RoutePrefix("api/test")]
public class TestController : ApiController { ... }
#endif
```

**Option 3: Require admin authorization**
```csharp
[RoutePrefix("api/test")]
[TokenAuthorize("Admin")]  // Restrict to admins only
public class TestController : ApiController { ... }
```
